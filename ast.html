About the Software Testing


7 Principles of Software Testing

Testing only can show that the defects are present, but cannot prove that there are no defects.
Testing everything (all combinations of inputs and preconditions) is not feasible except for trivial cases.
Testing activities should start as early as possible.
A small number of modules contains most of the defects.
One day the same set of test cases will no longer be able to find any new bugs.
Safety–critical software is tested differently from an ecommerce site.
If the system is unusable and does not fulfil the user’s needs and expectations then finding and fixing defects does not help.

    Equivalent Class Partitioning

    Idea: In your testsyou willuse just1 or 2 representativesfrom each group. Althoughthatdoesn’tguarantee that the rest of the values notcovered by the tests is bug-free, but usingseveralelements from the group will be quite illustrative.Keep in mind that both valid and invalid inputs can be divided intoclasses of equivalence and each class must have at least one representative used in testing.

    Boundary value analysis

    Idea: To select input variable values at their: Minimum; Just above the minimum; A nominal value; Just below the maximum; Maximum.

    Pairwise testing

    Idea: Createa list of values for each parameter and find out all unique pairs. Make sure your test cases cover each pair ofparameters.

    Decision tables

    Idea: The conditions areindicated as True/False or Yes/Novalues and the illogical combinations get reduced.

    State transition

    Idea: Use shapes to demonstrate the states and arrows to show the actions triggering the transition.
